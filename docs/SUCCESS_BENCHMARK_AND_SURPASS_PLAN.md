# 成功经验吸收与超越方案（API-first 版）

更新日期：2026-03-07

## 1. 当前判断

过去的对标研究已经完成，当前重点不是继续比较“谁的 OpenClaw 路径更完整”，而是把我方项目稳定收敛为：

- Python 核心业务引擎
- React 运营工作台
- Node webhook / proxy
- 闲管家 / Open Platform API 优先
- WS 或站点链路只做补充

## 2. 已经吸收的成功经验

### 2.1 低门槛启动

- 已提供 `start.sh` / `start.bat`
- 已提供 React / Node / Python 三服务启动口径
- 已提供前端首次配置引导

### 2.2 运行态可恢复

- 保留 Python 诊断、模块控制、日志读取能力
- Cookie 更新和服务控制已有真实接口入口

### 2.3 运营台产品化

- 工作台、商品、订单、自动上架、消息、配置、账号页面已进入主线
- 旧 code-review 壳层已清理

### 2.4 API-first 主路径

- 商品、订单、自动上架优先走闲管家接口
- 页面不再依赖 mock 数据或演示样本

## 3. 仍需超越的点

### P0

- 多店铺真实能力
- 消息链路状态统一
- 回调幂等和可观测性

### P1

- 运维面逐步从历史 Dashboard 迁到 React
- 错误提示继续非技术化
- 发布门禁和 release 说明持续标准化

### P2

- 进一步减少内部 `openclaw` 兼容命名
- 对 Legacy Browser Runtime 做更清晰的隔离

## 4. 超越目标

“超越”的定义不再是功能堆得更多，而是：

1. 主路径更简单，默认不依赖 OpenClaw。
2. 页面全部接真实接口，不用 mock 掩盖问题。
3. 配置和业务状态只有一个真相源。
4. 发布前检查和文档能被稳定复用。

## 5. 执行结论

当前项目已经完成“方向切换”，从多架构混用转为 API-first 主线。下一步要做的是继续补齐多店铺、消息状态和运维收敛，而不是重新引入旧的主路径依赖。
