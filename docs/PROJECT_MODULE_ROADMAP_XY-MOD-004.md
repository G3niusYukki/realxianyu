# 模块推进路线图（XY-MOD-004，API-first 主线版）

- 输出时间：2026-03-07 (Asia/Shanghai)
- 目标：围绕当前 `main`，继续完成 API-first 主线收口，而不是回到 OpenClaw 依赖架构

## 1. 目标与分批策略

### 1.1 总目标（4 周）

- `W1`：稳定配置 contract 和发布文档
- `W2`：补齐多店铺与账号映射
- `W3`：统一消息链路状态和回调观测
- `W4`：收敛运维面并形成可重复发布门禁

### 1.2 分批原则

1. 先主路径，后兼容层。
2. 先真实接口，后页面增强。
3. 先统一真相源，后扩展外围代理。

## 3. 里程碑

### Milestone 1

- 主文档、部署文档、API 文档、发布文档全部与当前 `main` 对齐。
- 建立标准发布检查命令和输出归档位置。

### Milestone 2

- 多店铺切换和账号映射具备真实能力。
- 店铺管理页面不再停留在单默认店铺视角。

### Milestone 3

- 消息链路状态可视化完成。
- webhook / callback 的失败、重试、幂等信息可追踪。

### Milestone 4

- 常用运维动作可从 React 页面完成。
- 发布前检查流程固定化，可直接复用。

## 4. 风险与缓解

| 风险 | 触发条件 | 缓解动作 |
|---|---|---|
| 配置真相源再次分裂 | Node / Python 各自新增写入口 | 所有写配置都回到 Python |
| 多店铺数据错绑 | Cookie 与闲管家店铺不一致 | 增加映射校验和页面告警 |
| 回调状态不稳定 | 外部平台字段漂移或乱序 | 固定回调 contract + 重试日志 |
| 运维入口重复 | React 和 Python Dashboard 长期分叉 | 明确迁移列表和下线条件 |

## 5. Definition of Done

- 有真实接口，不用 mock。
- 有错误路径和失败文案。
- 有测试或 smoke 证据。
- 文档同步更新。
- 不引入新的 OpenClaw 默认依赖。
